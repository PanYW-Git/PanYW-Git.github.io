问题记录

  • redis的hash类型的value是支持多种数据类型的，为什么会只能存入String？

    • 在使用stringRedisTemplate.opsForHash().putAll(tokenKey ,userDTOMap);导入redis时提示以下错误

      • Long无法转换为String

    • 看看chatgpt的回答，因为spring data redis为了统一操作，所有数据类型都需要是string，与redis hash类型的value其实并无关系！（其实前面讲过）

    • 因此需要在导入redis之前把map中的所有值转换为String类型，这个可以在bean转换为map的时候使用糊涂包的BeanUtil.beanToMap的重载方法

  • 登录成功后还是进入login界面

    • 这个是由于拦截器获取token时校验到的token为空，跳转到login.html页面

    • 找到问题就好解决了，发现是教程中登录成功反馈未反回token，返回就好了

缓存更新策略

  • 为什么需要缓存更新？

  • 因为数据同时保存在缓存和数据中，涉及数据一致性问题，如果对数据库数据做了一些修改缓存是不知道的，这种场景下会造成业务数据错误

  • 缓存更新策略：

    • 低一致性需求：比如店铺分类这种一般不会发送变化的场景
    • 高一致性需求：比如付款等场景，但主动更新还是需要加上超时作为兜底方案，因为不知道主动更新是否会失败，所以两种结合使用

  • 主动更新策略：

    • 01：自己更新，开发成本高，需要调用者自己更新数据库和缓存
    • 02：把数据库和缓存都整合为一个服务，调用者直接调用服务，但是这种方案如果服务一旦挂掉或者其他情况都会造成数据严重错误
    • 03：调用者只操作缓存，由其他线程异步将缓存持久化到数据库。但也涉及线程安全等问题会造成数据错误

  • 先删缓存，后操作数据库和先操作数据库后删缓存的多线程安全问题

    • 更新数据库涉及到写入磁盘操作，速度比较慢，因此第二种方案出现问题的概率要低一些（不是没有）

  • 缓存策略的最佳实践方案

缓存穿透的解决方案

  • 什么是缓存穿透？

    用户请求的数据再缓存和数据库中都不存在，这些请求都会传到数据库中，如果有恶意攻击一直访问不存在的资源那一直会反复的查询数据库

  • 解决方案

    • 缓存空对象（常用）：把null写到redis中

      • 优点：实现简单，方便维护

      • 缺点： 额外的内存消耗

        ​ 可能造成短期的数据不一致（可以通过新增的时候主动把数据放入redis中解决）

    • 布隆过滤

      • 优点：内存占用少，没有多余的key

      • 缺点：存在误判的可能，基于Hash算法，如果存在可能存在哈希冲突

        ​ 实现复杂

了解缓存雪崩

  • 缓存雪崩是指同一段时间大量的缓存key同时失效或者redis服务宕机，导致大量请求到达数据库，带来巨大压力
    • 解决方案：
      • 给不同的key的TTL添加随机值
      • 利用redis集群提高服务器的可用性
      • 给缓存业务降级限流策略
      • 给业务台南佳多级缓存

缓存击穿

Redis Double Check

  • Redis DoubleCheck:redis二次检查的作用是什么？

    • “Redis Double Check” 是一种设计模式，通常在使用 Redis 进行缓存时使用，用于解决缓存穿透问题。缓存穿透是指一个请求查询一个在缓存和数据库中都不存在的数据，导致该请求直接查询数据库，从而增加数据库的负载。

      “Double Check” 是指在查询缓存之前，首先进行一次检查，看看数据是否存在于缓存中，如果存在，则直接返回缓存数据。如果不存在，再进一步进行查询数据库的操作，并在查询到数据后，将数据存入缓存中，以供下一次查询使用。
